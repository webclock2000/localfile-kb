⚡ 直接输出 JSON 对象，不要输出任何分析、解释或思考过程。

从文档中提取知识图谱。返回一个 JSON 对象，key 为 "facts"，包含 0-15 条原子关系事实。
每条事实是一条边：(主语) --[谓语]--> (宾语) 三元组，并附带人类可读的上下文。
长文档（文章、论文、报告）通常应提取 8-15 条事实；短文档 0-3 条。

最重要的规则 —— 这决定了图的连通性：
主语和宾语必须是规范的**实体**或简短的原子**值**，绝不能是散文句子。
它们是图的节点：同一个实体每次都必须以完全相同的形式出现，这样边才能连接。
将叙述、证据和细微差别放在 description 中，不要放在 subject/object 中。

## 中文实体边界规则（极其重要）

实体名称必须是原子单位，不可拆分：
- 人名必须包含完整姓氏和名字（如"张三"不可拆为"张"和"三"）。
- 机构名必须包含完整后缀（如"华为技术有限公司"不可拆为"华为"和"技术有限公司"）。
- 复合姓氏必须保留（如"欧阳修"是一个人名，不可拆为"欧阳"和"修"）。
- 带称谓的人名保持完整（如"王教授"、"李总"不可拆为"王"和"教授"）。
- 职务+人名视为一个实体（如"首席执行官张伟"不可拆为"首席执行官"和"张伟"）。
- 地名包含完整行政区划后缀（如"北京市海淀区"不可拆为"北京"和"海淀区"）。

## 主语/宾语规则
- 使用实体的最短规范名称：人物、机构、产品/型号、数据集、方法、论文、地点、技术，
  或具体的原子值（日期、数字+单位、版本号、指标分数）。
- 去除冠词、角色和修饰语："华为公司" -> "华为技术有限公司"。
- 使用规范表面形式，不使用代词或改写。同一实体在所有事实中复用完全相同的字符串。
- 绝不将句子或从句放入 subject/object。
- 优先提取连接两个命名实体的关系（实体-实体边），这些边使图变得丰富。

## 每条事实格式
{
 "title": " ",
 "description": "<2-3句中文描述，包含答案+证据：实体、关系、值、日期/数字/来源详情。避免逐字重复标题。>",
 "subject": " ",
 "predicate": " ",
 "object": " ",
 "evidence_span": " ",
 "confidence": <0..100 整数>,
 "tags": [" ", ...]
}

## 覆盖优先级
当文本中有依据时，为以下每一项提取一条事实：
- 每个命名人物 + 其角色/职位/所属机构（即使只被提及一次）。
- 每个命名机构、产品、模型、数据集或方法 + 它与其他命名实体的关系。
- 每个具体日期/版本 + 它标记的事件或发布。
- 每个命名地点 + 在那里发生或位于那里的事物。
- 每个量化结果：指标分数、规模、速度提升、价格等。
- 每个跨实体关系：X 构建了 Y、X 基于 Y、X 与 Y 合作等。

## 反模式 —— 不要这样做
- 不要将描述性句子放入 subject 或 object。这会产生永远无法连接的孤立节点。
- 不要只提取主导实体的相关事实。只被提到一次的次要实体仍然值得拥有自己的边。
- 不要用通用套话填充预算。

## 谓语指导
- 始终为实际关系选择最精确的 snake_case 谓语（<=32 字符）。
- 以下仅为风格/粒度示例（不是固定列表）：built_by, based_on, trained_on, released_on,
  outperforms, evaluated_on, scored, integrates_with, acquired_by, authored_by,
  cites, position_held, works_at, graduated_from, located_in, founded_in,
  part_of, depends_on, developed_by, published_in
- 避免模糊的万能词如 affiliated_with 或 related_to。

## 标题和描述约束（违反者将被丢弃）
- title 和 description 必须读起来像自然的事实陈述。
- 它们不得提及文档、数据集或此任务。
- evidence_span 必须是上面提供的文档文本的逐字子串。
